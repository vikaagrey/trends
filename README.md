# trends - сервис трендов поисковых запросов

Читает поток событий из Kafka, считает Top-N уникальных поисковых запросов за последние 5 минут и отдаёт результат через HTTP API.

```
                        ┌─────────────────────────────────────────┐
                        │              trends-service              │
                        │                                          │
Kafka ──────────────►  Consumer                                    │
(search.events)         │  decode, validate, normalize             │
                        │  Add(query, source, ts)                  │
                        ▼                                          │
                       Engine  ◄── filter ── Stoplist (in-memory) │
                        │  HLL sliding window                      │
                        │  30 бакетов × ~10 с = 5 мин             │
                        │  Rebuild раз в 1 с                       │
                        ▼                                          │
                      Snapshot (atomic.Pointer)                    │
                        │  lock-free, готовый топ                  │
                        ▼                                          │
              GET /api/v1/top?n=N  ◄── клиенты (виджет)           │
                        │                                          │
              POST/DELETE /api/v1/stoplist                         │
                        │                                          └─────┐
                        ▼                                                │
                    Postgres  ◄─────────────── источник правды стоп-листа│
                        │  stop_words                                    │
                        │  (переживает рестарт)                          │
                        ▼                                                │
                     Redis ──── Pub/Sub ──► другие реплики              │
                        (опц.) сигнал "перечитай стоп-лист"             │
                                                                   └─────┘
```

Горячий путь `GET /top` не ходит ни в Kafka, ни в Postgres, ни в Redis — только читает готовый снапшот из памяти (2.5 нс, 0 аллокаций).

В реальном сервисе можно вынести горячий снапшот или checkpoint окна в Redis/Redis Cluster: это надёжнее при рестартах и проще для выравнивания реплик. В рамках задания топ хранится in-memory, потому что требовался быстрый Top-N за последние 5 минут без внешнего хранилища на read path.

## Запуск

```bash
make up    
make producer  # отправить 5 000 тестовых событий
```

Проверить результат:

```bash
curl "http://localhost:8080/api/v1/top?n=10" | jq .
curl http://localhost:8080/healthz
```

Остановить:

```bash
make down
```

## API

### GET /api/v1/top?n=10

Топ N запросов за последние 5 минут. `n` от 1 до 1000, по умолчанию 10.

```bash
curl "http://localhost:8080/api/v1/top?n=5" | jq .
```

```json
{
  "data": [
    {"query": "слайм", "count": 142},
    {"query": "sixseven", "count": 98}
  ],
  "timestamp": 1716480000,
  "window_sec": 300
}
```

`count` — число **уникальных** источников (`user_id` / `session_id`), не сырых кликов.

### Стоп-лист

```bash
# Посмотреть
curl http://localhost:8080/api/v1/stoplist | jq .

# Добавить слово (применяется на следующем rebuild, раз в 1 с)
curl -X POST http://localhost:8080/api/v1/stoplist \
  -H "Content-Type: application/json" -d '{"word":"спам"}' | jq .

# Удалить слово
curl -X DELETE http://localhost:8080/api/v1/stoplist/спам | jq .
```

### Healthcheck и метрики

```bash
curl http://localhost:8080/healthz   # {"status":"ok"}
curl http://localhost:8080/metrics   # Prometheus-метрики
```

### Ошибки API

| Сценарий | HTTP | Ответ |
|----------|-----:|-------|
| `n` не число или меньше 1 | 400 | Передайте нормальный `n`: `?n=10` |
| Стоп-слово пустое | 400 | Передайте `{"word":"..."}` |
| `POST /stoplist` без JSON-заголовка | 415 | Нужен `Content-Type: application/json` |
| Слово уже есть в стоп-листе | 409 | Сначала удалите его или используйте другое |
| Внутренняя ошибка сервиса | 500 | Детали остаются в логах, наружу не отдаются |

## Kafka-контракт

Topic: `search.events`, формат JSON:

```json
{
  "query":    "слайм",
  "source":   "user_abc123",
  "ts_ms":    1716480000000,
  "event_id": "evt_xyz789"
}
```

| Поле | Обязательно | Зачем нужно |
|------|:-----------:|-------------|
| `query` | да | Ключ агрегации. Именно поисковые запросы попадают в топ. |
| `source` | да | `user_id` или `session_id`. Метрика = число уникальных источников, не кликов — один бот с одного аккаунта не накрутит топ. |
| `ts_ms` | да | Время события на стороне поиска. Кладётся в бакет по времени события, а не обработки — лаг consumer не сдвигает окно. |
| `event_id` | нет | Для трассировки. Дубли безвредны: повтор того же `source` в HLL — no-op. |

Невалидные сообщения коммитятся и пропускаются — partition не блокируется (poison-pill safe).

## Конфигурация

Основные переменные:

| Переменная | По умолчанию | Зачем |
|------------|-------------|-------|
| `HTTP_ADDR` | `:8080` | Адрес HTTP-сервера |
| `POSTGRES_DSN` | `postgres://trends:trends@localhost:5432/trends?sslmode=disable` | Postgres со стоп-листом |
| `REDIS_URL` | `""` | Redis Pub/Sub для синхронизации стоп-листа между репликами |
| `KAFKA_BROKERS` | `localhost:9092` | Kafka brokers через запятую |
| `KAFKA_TOPIC` | `search.events` | Topic с событиями поиска |
| `KAFKA_GROUP_ID` | `trends-service` | Consumer group. Для полных реплик нужны разные group id |
| `WINDOW_SIZE` | `5m` | Размер окна топа |
| `BUCKET_COUNT` | `30` | Количество бакетов внутри окна |
| `TOP_K` | `1000` | Сколько позиций держать в снапшоте |
| `REBUILD_INTERVAL` | `1s` | Как часто пересобирать снапшот |

Дополнительно есть `REQUEST_TIMEOUT`, `SHUTDOWN_TIMEOUT`, `LOG_LEVEL`.

## Архитектура

### Почему hopping window (30 бакетов), а не точное sliding window

5 минут делятся на 30 бакетов по примерно 10 секунд. Запись события идёт в один текущий бакет за O(1). Rebuild объединяет все живые бакеты и сортирует.

Погрешность границы окна — примерно 10 с. Для виджета трендов это допустимо, точное посекундное окно стоило бы на порядок дороже по памяти и CPU.

### Почему HyperLogLog, а не Set

Нам нужно считать не все повторы запроса, а сколько разных `source` его сделали. Например, если один пользователь 100 раз ищет `слайм`, в топе это должно считаться как `1`, а не как `100`.

Самый простой вариант — хранить `Set` всех `source` для каждого запроса. Он точный, но быстро раздувается по памяти: популярный запрос может собрать тысячи пользователей в каждом бакете.

HyperLogLog хранит не сами `source`, а компактный счётчик уникальности. Он даёт небольшую погрешность, зато память почти не растёт вместе с числом пользователей. Для топа трендов это нормальный обмен: нам важен порядок популярных запросов, а не абсолютно точная бухгалтерия до одного пользователя.

### Почему каждая реплика читает весь топик

Каждая реплика использует уникальный `KAFKA_GROUP_ID` и независимо строит полный локальный топ. `GET /top` не требует координации между репликами. Цена - дублирование чтения Kafka на N реплик.

Дополнительный trade-off при мульти-репликах: реплики стартуют в разное время, поэтому первые `WINDOW_SIZE` (~5 мин) после старта каждая отдаёт разный топ (у одной окно уже заполнено, у другой ещё нет). Для виджета это кратковременное расхождение — приемлемо.

### Почему стоп-лист в Postgres + Redis

Стоп-лист должен переживать рестарт, поэтому он хранится в Postgres.

В памяти сервис держит только быстрый кэш этого списка. Когда слово добавляют или удаляют через API, сервис сначала пишет изменение в Postgres, потом перечитывает список в память.

Redis нужен только для нескольких реплик: одна реплика изменила стоп-лист и отправила сигнал, остальные получили сигнал и тоже перечитали список из Postgres. Если Redis временно недоступен, запись в Postgres всё равно остаётся успешной.

Изменение стоп-листа может попасть в топ не мгновенно, а на следующей пересборке снапшота. По умолчанию это до 1 секунды.

## Trade-offs и проблемы постановки

### Неточности в задании и принятые решения

| Проблема в постановке | Как решено |
|-----------------------|------------|
| «Скользящее окно» — тип не уточнён | Hopping window: 30 бакетов × ~10 с. Погрешность границы около 10 с — допустимо для виджета трендов. Точное sliding window дороже на порядок. |
| «Анти-накрутка» — механизм не описан | `count` = число уникальных `source`, не сырых кликов. Один бот с одного аккаунта повторяет запрос 1000 раз → `count = 1`. Реализовано через HyperLogLog. |
| «Поток сырых логов» — формат не задан | Сформирован контракт: `query`, `source`, `ts_ms`, `event_id`. Обоснование — в разделе «Kafka-контракт». |
| «Самые популярные» — по числу кликов или уникальных пользователей? | По уникальным `source`: релевантнее для реального тренда, защищает от накрутки. |
| «Несколько реплик» — как синхронизировать стоп-лист? | Redis Pub/Sub: при изменении стоп-листа через API реплика публикует сигнал, остальные перечитывают из Postgres. Без Redis — каждая реплика обновляет только свой кэш. |
| Как ведут себя повторные изменения стоп-листа? | `POST` уже существующего слова возвращает `409 Conflict`, потому что конфликт определяется Postgres. `DELETE` идемпотентен: удаление отсутствующего слова возвращает успех. |
| Нормализация query не определена | `trim` + `lowercase` + схлопывание пробелов + ограничение 256 символов. Без этого `"SixSeven"` и `"sixseven"` — разные строки в топе. |
| Синонимы и сокращения не определены | Сейчас `"попит"` и `"поп ит"` останутся разными строками в топе. Для объединения нужен отдельный словарь канонических запросов: например, `попит -> поп ит`, `лизун -> слайм`. Это отдельный слой нормализации, который можно хранить в Postgres и держать в in-memory cache по аналогии со стоп-листом. Автоматически склеивать такие запросы опасно: сокращения бывают неоднозначными. |
| «Холодный старт» — что показывать после рестарта? | Окно пустеет, `GET /top` возвращает `[]`. Прогрев занимает `WINDOW_SIZE` (~5 мин). Это ожидаемое поведение, документировано. |

### Компромиссы ради производительности

| Компромисс | Что получаем | Цена |
|------------|-------------|------|
| In-memory окно + atomic снапшот | `GET /top` = 2.5 нс, 0 аллокаций, без БД/Redis на read path | При рестарте окно прогревается заново ~5 мин. В более надёжном варианте можно добавить Redis checkpoint |
| Hopping window (бакеты) | O(1) запись, O(K) rebuild | Граница окна неточная ±`bucket_size` |
| HyperLogLog вместо точного Set | До ~16 КБ на счётчик вместо сотен КБ для точного Set | Погрешность count ~2% |
| Rebuild раз в секунду | Дешёвый `GET /top` без пересчёта | Новые события и стоп-лист видны с задержкой до 1 с |
| Каждая реплика читает весь топик | Полный локальный топ, нет координации | CPU × N реплик на чтение Kafka |

## Метрики Prometheus

| Метрика | Тип | Описание |
|---------|-----|----------|
| `trends_events_consumed_total` | counter | Записано в окно |
| `trends_events_decoded_total` | counter | Успешно распарсено (до фильтрации окном) |
| `trends_events_invalid_total` | counter | Битые / невалидные |
| `trends_events_norm_dropped_total` | counter | Пустые после нормализации |
| `trends_events_out_of_window_total` | counter | Вне временного окна |
| `trends_consumer_retries_total` | counter | Перезапуски consumer после ошибок Kafka |
| `trends_rebuild_duration_seconds` | histogram | Время пересборки снапшота |
| `trends_snapshot_size` | gauge | Уникальных queries в снапшоте |
| `trends_stoplist_words_total` | gauge | Слов в стоп-листе |
| `trends_http_request_duration_seconds` | histogram | Latency хендлеров (метки: method, path, status) |

Минимальный мониторинг:

- scrape `GET /metrics`;
- алертить рост `trends_consumer_retries_total` — Kafka недоступна или consumer постоянно падает;
- смотреть p95/p99 `trends_http_request_duration_seconds` по `path="/api/v1/top"`;
- смотреть `trends_rebuild_duration_seconds`: если rebuild стабильно дольше `REBUILD_INTERVAL`, снапшот начинает отставать;
- `trends_snapshot_size` показывает размер готового топа, `trends_stoplist_words_total` — текущий размер стоп-листа.
